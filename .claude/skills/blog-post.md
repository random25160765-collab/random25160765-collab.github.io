# Blog Post Review Skill

在文章发布前，从内容质量角度做最终审阅。本 skill **不涉及格式规范**（格式问题交给 `blog-format`），只关心一件事：**读者能不能用最少的精力，获得作者最想让 ta 理解的东西。**

## 核心原则

### 1. 准确 > 花哨

先把问题讲清楚，再谈形式。如果一段文字里概念模糊、因果关系跳跃、术语没有解释，换什么格式都没用。**准确是底线。**

### 2. 减法 > 加法

博客润色最容易犯的错是"加东西"——加图、加表格、加动画。但读者的注意力是稀缺资源。**删掉一句废话比加一个组件有价值十倍。** 真正好的润色往往是在做减法：砍掉偏离主线的分支、合并重复的表达、把绕弯子的句子捋直。

### 3. 结构服务于叙事

目录让读者预览全局；callout 把"注意/警告/思考"跟正文拉开层次；图表在文字难以表达的地方（架构、流程、时序、数据对比）替代文字。这些手段的衡量标准只有一个：**让读者的认知负担降到最低。** 如果一段纯文字已经讲清楚了，硬上图表或组件就是画蛇添足。

### 4. 读者是否能"带走"

读完这篇文章，读者带走的是"知道了作者做了什么"，还是"觉得自己也能做类似的事"？如果是前者，说明推导过程被压缩掉了——你给了结论，没给"为什么是这个结论"。

## 审阅流程

### 阶段 0：建立基准

- 通读目标文章，用一句话概括文章试图回答什么问题
- 确认文章的目标读者是谁（同行？新手？未来的自己？）
- 如果用户提供了同主题参考文章，阅读并对比

### 阶段 1：准确性检查

逐段检查以下问题：

1. **概念是否清晰**：术语第一次出现时有没有解释？容易混淆的概念对（如模拟 vs 真实、桥接 vs 实现）有没有明确区分？
2. **因果关系是否完整**：每个"问题→对策→原理"的逻辑链有没有断点？如果写了"因为 X，所以 Y"，中间跳过的推理步骤读者能自己补上吗？
3. **代码是否正确**：代码块与真实源文件是否一致？禁止"美化"——省略错误处理、边界检查会让读者误判代码质量。
4. **信息是否可验证**：引用的外部数据、版本号、仓库链接是否准确？

### 阶段 2：减法扫描

这是最关键的阶段。逐段审视：

1. **偏离主线的分支**：这段话如果删掉，会影响读者理解文章的核心论点吗？如果不会，它就是噪音。
2. **重复表达**：同一个概念在不同小节里被反复解释？合并它们。
3. **绕弯的句子**：把这个长句改成两个短句，意思变了吗？如果没变，就拆开。
4. **炫技式引用**：引用的外部资料/论文/链接，是读者理解本文所必需的，还是只是"显得有广度"？

> [!TIP] 减法检查的核心问题
> 对每一段文字，问自己：**"如果这段删掉，读者会 miss 掉什么？"** 如果答案是"不会 miss 什么重要的"，就删。

### 阶段 3：结构审查

1. **时间线 vs 概念层**：文章是按"做了什么→又做了什么"组织的，还是按"回答什么问题"组织的？时间线适合复盘记录，概念层适合教学。如果目标读者是"想做类似事情的人"，建议按概念层重组。
2. **小节自检**：每个小节能否用一句"这一节回答什么问题"概括？如果不能，说明该小节缺乏明确的探问主线。
3. **散点聚合**：如果文章列举了多个独立技巧/优化/结论，它们之间有没有层次关系？能否拼成一条逻辑链？如果不能，需要在开头或结尾解释"这些手段之间的关系是什么"。
4. **收束句**：每个关键小节末尾是否有一句话把具体内容收敛到通用原则？让读者在"知道了细节"的同时"建立了模型"。
5. **目录**：短文章（< 3 个小节）不一定需要 `## Table of contents`；长文章（≥ 5 个小节）建议有。

### 阶段 4：叙事节奏

1. **试错 vs 直接给结论**：文章里的关键决策点，读者是经历了"试了 X → 不行 → 为什么不行 → 选了 Y"，还是直接收到了"问题：X，对策：Y"？前者让读者积累判断力，后者只传递信息。**至少有一个关键决策点应该保留试错过程。**
2. **"为什么"密度**：全文有多少处解释了"为什么选这个方案而不是别的"？每个关键设计分叉点至少需要一句"为什么"。
3. **阅读疲劳点**：连续多少段纯文字没有换气口（图表/callout/代码块/小节标题）？如果超过 5 段纯文字连续出现，读者可能感到疲劳。

### 阶段 5：全局审视

1. **一句话总结**：读完文章后，能不能用一句话说出它的核心价值？这句话文章里有没有直接或间接地表达出来？
2. **标题是否诚实**：标题承诺的内容，文章是否覆盖了？反过来，文章真正的核心内容，标题是否暗示了？
3. **开头和结尾**：开头是否让读者知道这篇文章"适合谁、讲什么"？结尾是否把内容收敛到通用原则，而不是"就写到这"？

## 输出格式

审阅结果按以下结构输出：

```
## 审阅结果：[文章标题]

### 一句话概括
（这篇文章试图回答什么核心问题）

### 必须修改
（准确性问题——概念错误、逻辑断点、代码不一致）

### 建议删除
（偏离主线、重复表达、噪音——逐条说明删掉后读者会 miss 什么）

### 结构建议
（组织结构、收束句、小节划分——逐条说明当前问题及建议方案）

### 叙事节奏建议
（试错过程、为什么密度、阅读疲劳点——逐条说明）

### 全局收束
（标题、开头结尾、一句话核心价值）
```

## 注意事项

- **只给具体建议，不给泛泛评价**：不能说"这里写得不好"，要说"这里跳过了'为什么选 Y 而不是 Z'的推导，建议加一句 XX"。
- **减法优先于加法**：默认立场是"删"，只有确实影响理解时才建议"加"。
- **尊重作者语气**：不强行"润色"作者的表达风格，只纠正确实影响理解的地方。
- **不干预格式**：frontmatter、代码块语法、callout 格式等问题交给 `blog-format`，本 skill 只关心内容质量。
- **审查只读**：输出建议，不要直接改文章。让作者自己决定改哪些。
